Assistant Factory · An Engineering Thesis

Most agents are a prompt
and a while loop.

Ours are built around explicit beliefs — what the agent currently thinks, what it remembers, and what it knows it doesn't know. Tracked structurally, not implicitly through a context window.

Prompt engineering is a leaky abstraction. The bigger your system, the worse the leak gets. We re-introduced the boring stuff — symbolic state, structured dialogue, calibrated uncertainty — and let the LLM be a component, not the whole brain.

§ 01

Where the while loop hits a wall

The honest pitch

Every agent SDK is a thin wrapper around the same loop: read input, shove it back into a prompt with the tool list and the history, parse a tool call, run, repeat. It works for one turn. It mostly works for two. By turn five, the model is reconstructing state by re-reading the transcript every call.

That's not state. That's a recap, performed by the LLM, every single time. It's expensive, it's fragile, and it has no opinion about what's actually happening.

The cost shows up everywhere. Drift across long sessions. Hallucinated history. Confident bad answers because the model can't tell you it isn't sure. Debugging that turns into archaeology on a 30k-token blob. Prompts don't scale. Beliefs do.

If you remember one thing

Prompts encode behavior implicitly, in the model's attention. Beliefs encode behavior explicitly, in structured state.

Failure modes that vanish
  • "It forgot what we agreed five turns ago."
  • "It's confidently wrong and won't say it isn't sure."
  • "I can't tell why it picked that tool."
  • "It re-asks for the same thing."
§ 02

Three things we track explicitly

The pillars

Each one is a structured object with an API, not a paragraph in a prompt. The LLM reads from these and writes back to them. It's the brain in a body, not a brain in a box.

A · belief
What we think we're doing right now.

The current goal, the entities in play, what's filled in, what's still missing. A real object the agent updates each turn.

# dialogue state, abridged
state.goal = "rework post"
state.entity = "three-hour-marathon"
state.scope = None # missing
B · memory
What we remember, layered by horizon.

Three tiers: a session scratchpad for what we just learned, a per-account preferences store, and slow-moving context about your team and your work. Different reads. Different writes.

# memory tiers
memory.session.read()
memory.prefs.get("tone")
memory.context.retrieve(q)
C · uncertainty
What we don't know — and at what level.

Calibrated, declared, addressable. The agent rates how unsure it is about each part of the state, and that rating decides whether to ask, confirm, or proceed. Not vibes — a number.

# declared uncertainty
amb.declare("specific",
  slot="scope")
amb.action == "ask"
The synthesis

"Belief, memory, and uncertainty are first-class objects. The LLM is a tool that reads and writes them — not the system itself."

Why this is the unlock

Once these three are objects, every interesting agent property becomes implementable: multi-turn coherence, calibrated abstention, deterministic recovery, debuggable failures. They were unreachable when state lived inside a prompt.

§ 03

What this buys you

The payoff
01 · debuggability

A failure has a place to land.

When something goes wrong, you read the state — not a 30k-token transcript. You can write a unit test against the state.

02 · recovery

Failures don't compound.

If a tool returns garbage, the policy can retry, escalate, or re-route — because the state is intact. The agent doesn't unravel the way a one-shot prompt does.

03 · calibration

It can tell you it isn't sure.

Because uncertainty is a tracked object, the agent has a structural way to express it — to you, or to itself before acting. No more confidently-wrong answers as a default mode.
