Today we're doing a deep-clean codebase review prior to public release. Your priorities, in order:

Functional bugs: Anything that could cause incorrect behavior users would encounter — problems with sending, receiving, or displaying messages, ambiguity handling, flow and slot interaction, dialogue state synchronization across turns, utterance ordering, protocol mismatches, or silent data loss. Trace real data flows end-to-end rather than trusting function names or comments. Don't believe something works just because the code looks reasonable; verify that the actual logic handles edge cases, error paths, and boundary conditions.

"Good enough" code hiding real problems: Look hard for places where the code mostly works but isn't doing the right thing. This includes: safe fallbacks or default values that silently mask upstream errors instead of surfacing them; retry loops, polling, or spin-waits that paper over timing or synchronization issues rather than solving them; performance costs accepted to avoid getting the async flow right; and broad try/catch blocks that swallow exceptions to keep things moving. This is a codebase dealing with async tool-calling and text streaming, so some amount of waiting and retry is inherent — but each instance should be critically evaluated. When running each skill to complete the task, is each extra tool-call necessary, or is it compensating for a missing slot, a dropped parameter, or an incorrectly predicted flow? Would more exemplars, a proper explanation, or a more precise output format be clearer or more performant? The most dangerous code in a codebase like this isn't code that fails, but code that quietly degrades until the user is miserable.

Missing or inadequate error handling: Places where failures are swallowed, where invalid slot-values propagate silently, or where a network/protocol hiccup could leave the system in an inconsistent state. If a design choice is actively creating bugs, making them easy to introduce, or harder to fix, call it out.

Architecture issues: duplicated code, unused functions, split responsibilities, and code anti-patterns that could be rectified by responsibility unification, better organization, or separation of concerns. We already have a canonical set of components for the majority of the work. Are we accidentally re-inventing the wheel by adding a function which is already handled by an existing component?

Missing tests for critical paths: Core functionality that has no test coverage, or where existing tests are so heavily mocked they wouldn't catch a real regression. 

Documentation drift: Errors or stale information in the primary or secondary AGENTS.md files, README.md, inline code comments, docker-compose files, Dockerfiles, environment/config examples, and any other documentation a developer or user would rely on to set up, run, or understand the project. If the docs describe behavior the code doesn't actually implement, or omit steps/config that are actually required, flag it. These are release blockers — someone trying to follow the README on day one will hit them immediately.
