## _hidden_section_title
Training dialogue agents for real-life use cases is immensely difficult — manual data annotation quickly hits scaling limits. One solution is to build a user simulator that generates training examples for the agent at scale. To build such a system, though, you need a model that understands how to react to agents; yet to train that model, you need a dialogue agent to interact with. The problem is a classic chicken-and-egg, but it comes with one important asymmetry. That asymmetry runs along two dimensions: pipeline complexity and scope. On complexity: a dialogue agent must ingest natural language, execute a policy (often with API calls), and return a natural language response. The simulator skips the NLU step entirely and moves straight into policy management. On scope: the range of user actions is bounded by whatever APIs the product exposes. A real user can do countless things, but the agent is only expected to handle a small, well-defined subset — for everything else, it can abstain. The two dimensions push the simulator's design in opposite directions. Complexity calls for a system capable of handling sophisticated actions and paired with a strong NLG component — a profile that already fits an existing model class: open-domain chatbots. Task-oriented deployments routinely encounter users who break from script and drift into chit-chat; a simulator that mirrors that behavior would make the resulting agent more robust by default. Scope pulls the other way, keeping the policy lean. Rather than learning from interaction, we define a rule set that enumerates all combinations of user intents, then sample from it to construct plausible scenarios. Take the flight domain as an example: a user may want to book a flight, cancel a flight, check on an existing reservation, or ask about FAQs. In practice there may be a dozen more actions, but not many more than that. A bounded action space says nothing about expressive complexity, though — meaning the number of values users provide and how they phrase them. When booking a flight, a user may choose from thousands of departure and destination locations and times — millions of possible values, yet only four slot combinations. Accounting for edge cases like price requirements, seating preferences, and airline choices, that might expand to around 100 slots for a given domain. Even so, the number of distinct slots stays small relative to the number of values — there are a hundred ways to say "Fly from San Diego to Miami on Saturday morning," yet every variation shares the same underlying structure.[^1] The complexity lives entirely in how a request is expressed, not in its form. That gap — a finite slot set against an almost infinite value space — is precisely what makes the design tractable. With domains, slots, and intents bounded, the policy can scale to any combination without friction; the heavy lifting falls entirely to the text generation layer. The natural candidate is the open-domain chatbot: these models already query the internet and digest millions of Wikipedia pages, so parsing a few dozen parameters before producing a natural language response is well within their capabilities.
