---
title: "Using Multi-modal Models to Improve AI Agents"
---

## Motivation
Most AI agents today run on text alone. That leaves them blind to screenshots, diagrams, and the UI elements humans rely on every day. Multi-modal models close that gap by adding vision, so agents can perceive their environment the way people do. That shift matters because the tasks we most want to automate — web navigation, document analysis, GUI control — are visual by nature. Strong vision-language models are now widely available, so this is no longer a research problem. Real-world adoption is underway.
## Ideas
One promising direction is hierarchical vision: the agent first builds a coarse understanding of the overall scene, then zooms into fine-grained regions only when a step requires it. This reduces unnecessary computation while keeping precision high where it counts. A related approach is to treat the vision model as a callable tool rather than an always-on encoder, so the agent invokes it selectively and falls back to text-only processing when the latency budget is tight. On the data side, synthetic environments can generate large volumes of rendered screenshots for fine-tuning the vision-action policy without the cost of manual annotation. Incorporating video frames instead of individual screenshots also enables temporally-aware agents that track state changes across steps rather than treating each observation as isolated. These ideas set the strategic direction; translating them into a working system means addressing each layer of the implementation in turn.
## Process
With those directions in mind, the practical question becomes how to actually build the pipeline. Building a multi-modal agent starts with selecting the right vision encoder — CLIP and SigLIP are strong candidates — and pairing it with a language backbone that drives the planning loop. Before any pixels reach the model, visual inputs need preprocessing: screenshot cropping to isolate relevant regions, OCR augmentation to surface embedded text, and bounding-box annotation to give the model spatial anchors. With inputs ready, visual embeddings are fused with text context and passed to the action-selection model. Concretely, the vision encoder converts each screenshot into a fixed-size embedding vector that encodes spatial layout and visual semantics; this vector is projected into the language model's token space via a learned linear layer, then prepended to the text context tokens so the planner attends over both modalities simultaneously when selecting its next action. The action space itself must be defined in terms the vision model understands, referencing visual coordinates or named UI components rather than abstract token sequences. Once the pipeline is wired together, evaluation should cover both task-completion metrics and visual-grounding accuracy — catching failures that text-only benchmarks would miss entirely. When performance is satisfactory on held-out workflows, the feature can be shipped safely behind a flag while the team monitors production behavior.
## Takeaways
Adding multi-modal grounding meaningfully reduces agent errors on tasks that involve complex visual interfaces — the improvement is real and measurable. That said, it comes with a cost: both latency and inference spend increase, so teams should think carefully about when to invoke the vision encoder and when skipping it is the right call.

In production settings, fine-tuning on domain-specific visuals consistently outperforms zero-shot vision, making labeled examples from your own UI a worthwhile investment. Ultimately, the biggest gains come not from bolting vision onto an existing agent as an afterthought, but from tightly coupling visual observations with the reasoning step so the model can act on what it sees in one coherent flow.
