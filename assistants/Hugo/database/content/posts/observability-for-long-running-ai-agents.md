---
title: "Observability for Long-Running AI Agents"
---

## Motivation
Long-running agents can drift silently over hours — small prompt deviations compound into large behavioral shifts with no obvious signal. Without real-time observability, failures only surface at task completion, if they surface at all.

The work described here was sparked by a concrete incident: a weekend agent run that produced hundreds of unexpected API calls and left no audit trail, resulting in a billing surprise on Monday morning. That experience made the case for treating AI agents like production services — instrumented with metrics, logs, and alerts from day one.
## Latency Targets
Agents don't fail at the median — they fail at the p99, where a single slow tool call compounds across hundreds of turns and blows end-to-end task duration past any reasonable ceiling. Per-step latency and end-to-end duration are related but distinct, and you need SLOs for both before instrumentation can tell you anything useful. Establishing per-step SLOs for tool calls, LLM inference, memory retrieval, and orchestration overhead gives you the scaffolding to catch those compounding hotspots early. In practice, a useful latency budget tracks three numbers: the p50 and p99 per turn, a full end-to-end versus per-tool-call breakdown, and a hard ceiling that automatically triggers a kill switch when breached. Async checkpointing is a key strategy for staying within that budget — by avoiding blocking on slow sub-tasks, the agent can make forward progress while expensive operations resolve in the background.
## Token Accounting
Token accounting means tracking input and output tokens at every LLM call — not just the final response. Context windows grow across hundreds of turns, and each step's token cost quietly inflates with them. Tracking usage per tool, sub-task, and agent role surfaces that growth before it becomes a problem. The three metrics that matter most in practice are per-turn input tokens, cache-hit rate, and tool-result tokens — the last of which often dominates total usage and is the easiest to overlook. Setting per-run token budgets and emitting warnings before hard limits are hit converts a silent risk into an actionable signal.
## Cost Modeling
Token counts only become actionable when they are translated into dollar costs in real time, not post-hoc. Mapping those costs to specific task types quickly reveals which agent behaviors are expensive and which are cheap, a distinction that is nearly impossible to make from raw token logs alone. The stakes became clear one weekend when a batch agent was left running Friday evening to process a backlog of low-priority summarization tasks. Because cost anomaly detection was not yet wired up, nobody saw the alerts that would have fired had they existed. By Sunday night, the agent had burned through nearly three times its expected budget. A subtle prompt-construction bug had caused it to re-fetch and re-summarize overlapping document windows on every iteration, tripling token consumption per task. The damage only surfaced Monday morning when the billing dashboard refreshed. That experience put a hard requirement on the roadmap: real-time cost anomaly detection that fires when a run trends beyond its budget envelope, long before the billing cycle closes. Rolling cost projections were the next addition. By combining current spend-per-task with historical completion rates, the system can now estimate remaining cost at any point mid-run and surface a projected total before the work is done. That visibility gave the team the ability to start reasoning about sampling strategy in economic terms. The headline metric that unlocked that reasoning is cost-per-completed-task: spend tied directly to value delivered, not to tokens consumed in the abstract. When the team plotted this metric across different sampling temperatures and reasoning depths, the results were striking. High-temperature sampling and extended chain-of-thought passes inflated token counts significantly, but the improvement in task completion rate was modest. Dialing temperature down from 1.0 to 0.6 and capping reasoning steps at four reduced cost-per-completed-task by roughly 30 percent, with only a marginal drop in output quality. Cost observability had moved from a monitoring concern to an active input into model configuration decisions.
## Dashboards
A practical observability dashboard for long-running agents surfaces five key panels: active agents, step throughput, cumulative token spend, error rate, and task completion percentage. Alongside the aggregate view, a live trace panel — rendered as a waterfall chart of tool calls and LLM requests for a single run — makes it possible to debug a specific agent's behavior without digging through raw logs. Aggregated cost heatmaps, sliced by agent type, model, and time window, reveal spending patterns that per-run views obscure. Alert configuration ties everything together: thresholds for runaway spend, stalled tasks, and repeated tool failures ensure that the dashboard is not just a monitoring tool but an early-warning system. Latency-percentile views — tracking p50, p95, and p99 per tool call and per agent turn — expose tail-latency outliers that average-based panels routinely hide. Per-tool token panels break down cumulative token spend by individual tool, making it straightforward to identify which integrations are the heaviest consumers and where prompt trimming or caching would have the greatest impact.
## Takeaways
Observability is not optional for production AI agents — silent drift and surprise costs are the default without it. The right place to start is token accounting and latency targets; those two pillars surface the most common failure modes before adding dashboard complexity on top.

A lightweight instrumentation layer pays for itself after the first incident, and usually well before. The recommended next steps are practical and low-lift: adopt structured logging, wire up a cost alert, and commit to reviewing traces on a weekly cadence. Each step builds the operational muscle that keeps long-running agents from becoming long-running liabilities.
