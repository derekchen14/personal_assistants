# SWE2 implementation plan — round 5.0

Round 5.0 adds one new concept — an NLU thread handle passed into PEX plus an idempotent `_settle_nlu()` join — and names the two hooks (pre-tool on read_state, pre-flow on activate) where the join fires. The parallel-think path in `agent._orchestrate` already builds a `thread`; today it's only joined at the turn boundary (agent.py:84-85), so the acting loop reads LAST turn's belief. The fix: hand `thread` to `pex.execute`, store it for the turn, and call `_settle_nlu()` the first time the loop consumes belief — on `read_state` (the primary Plan-turn path, which the prompt already tells Plan turns to hit first) and as a backstop before any flow activates. `_settle_nlu` joins once then clears the handle, so all later calls (including the turn-boundary join in agent.py) are no-ops. Non-Plan/awaited/click turns pass `thread=None` and are unchanged. No hook registry, no method renames — the six hook points stay as the existing inline hooks; only the pre-tool and pre-flow hooks gain the join line. I flag one gap the spec's literal wording misses: `_dispatch_write_state`'s stackon+active branch folds belief's `pred_slots` via `_apply_belief_slots` BEFORE it routes into `activate_flow`, so a stackon+active turn that skipped read_state would fold STALE slots even though activate_flow later settles belief for the policy run — the clean fix adds the same idempotent `_settle_nlu()` call at the head of that branch (a third call site).

## Risks

Main risk is the `_apply_belief_slots` ordering gap: the spec literally names pre-flow = activate_flow (pex.py:618), but `_dispatch_write_state`'s stackon+active branch calls `_apply_belief_slots` (reads state.pred_flows/pred_slots) BEFORE routing into activate_flow. If the join lives only in activate_flow, a stackon+active turn that skipped read_state would fold LAST turn's slots into the flow entry even though the policy run then sees fresh belief — a silent stale-slot bug that partly defeats the fix. Mitigation: add the same idempotent `_settle_nlu()` at the head of that branch (change (f)); it's a no-op on the common path where read_state already settled. I flag it rather than hide it because SWE2's charter is to surface where the minimal-diff path accrues a special-case. Second risk: leaving the turn-boundary `thread.join()` in agent.py after the handle is cleared — verified harmless: `_settle_nlu` sets the PEX handle to None but agent.py's local `thread` var still references the (already-finished) Thread, and Thread.join() on a completed thread returns immediately. Third: the test's 50ms sleep proves ordering deterministically (read_state must block on join or it reads the pre-write default None); it's a real thread but bounded and cheap, within the eval-speed doctrine. Idempotency of `_settle_nlu` (guard on truthy handle + clear) means the three call sites never double-join. No behavior change for click, awaited-think, or plain-reply turns (all pass thread=None → _settle_nlu is a no-op), so the existing orch_agent/mock_agent suites stay green.
